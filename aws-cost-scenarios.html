Arquitectura del sistema

Signal Catcher es un pipeline de ad tracking en tiempo casi real: la API acepta eventos de impresión, click y conversión, los encola de inmediato (202 Accepted) y los workers procesan de forma asíncrona, escribiendo métricas agregadas en InfluxDB y el JSON crudo en almacenamiento tipo objeto (MinIO local, S3 en AWS). Grafana consulta InfluxDB para cuadros de mando y reconciliación bajo carga. Todo el stack corre hoy con Docker Compose (un solo docker compose up).

InfluxDB vs MinIO
InfluxDB guarda puntos de serie (tags: tipo, estado, anunciante, campaña, ad…; campos: conteos, revenue, tiempos) para agregados rápidos y paneles.
MinIO guarda el payload completo por partición horaria para auditoría, reprocesos o consultas tipo objeto — no sustituye al TSDB.
Por qué RabbitMQ (y no solo HTTP interno)
Desacopla la velocidad del cliente (202 rápido) de la velocidad de ingestión a Influx/S3. Si los workers van lentos, la cola absorbe el pico sin tumbar la API.

Retención (resumen)

Tier Retención Racional breve
InfluxDB 30 días (init Docker) Dashboards usan ventanas ≤ 24 h; evita crecimiento ilimitado en disco.
MinIO / S3 Política de ciclo de vida (p. ej. 90 días + frío) Auditoría y re-procesamiento; en AWS se combina con Glacier / Intelligent-Tiering.
DLQ Hasta revisión manual Eventos fallidos no se descartan en silencio; operación puede reinyectar o analizar.

Decisiones que suelen preguntar en defensa

  • Cola: RabbitMQ con exchange direct y una cola por tipo de evento; DLX + DLQ. Kafka aportaría más complejidad operativa para un patrón “procesar una vez por mensaje”.
  • TSDB: InfluxDB 2.x por modelo tag/field flexible y Flux; Timescale implica Postgres + migraciones; Prometheus es más pull/métricas de infra que ingesta masiva de eventos de negocio.
  • Objeto: MinIO en laboratorio por paridad S3 sin credenciales AWS; en nube → Amazon S3 con el mismo contrato de particiones.
  • Escalado: más réplicas de api y consumer en Compose; en AWS más tareas Fargate / Auto Scaling detrás del mismo ALB y colas compartidas.

Cómo reducirlo aún más
El NAT Gateway cuesta $33/mes solo de tarifa fija, independiente del tráfico. Para este escenario conviene usar VPC Endpoints para S3 y ECR (gratuitos o $7/mes), y sacar los servicios a una subred pública si el modelo de seguridad lo permite. Ahorro potencial: ~$26/mes. Con Savings Plan de cómputo al 1 año: reduce compute en ~22%, ahorro adicional de ~$8/mes.
⚠️
Riesgo del tier bajo: RabbitMQ self-managed en t3.medium no tiene SLA ni failover automático. Si la instancia muere, los mensajes en tránsito se pueden perder (aunque los durable queues sobreviven reinicios). Para producción real, migrar a Amazon MQ o usar un volumen EBS separado con snapshots automáticos.

Producto en producción con tráfico real moderado

Amazon MQ domina el costo con $210/mes aunque el broker esté desocupado el 80% del tiempo. Cómo reducirlo
Con Compute Savings Plan 1 año en ECS Fargate: el compute baja de $108 a ~$84 (ahorro 22%). Mover InfluxDB a Reserved Instance m5.large 1 año reduce la instancia frente al on-demand.
